# 모바일 가위바위보 로또 웹앱 개발 준비 체크리스트

## 1. 문서 목적

이 문서는 `닉네임 + 4자리 PIN` 로그인 기반의 모바일 가위바위보 로또 티켓 웹앱을 개발하기 전에 미리 준비해야 할 항목을 정리한 체크리스트다.  
기준 스택은 `Next.js + Supabase + GitHub + Cloudflare Pages`다.

---

## 2. 전체 준비 순서 요약

아래 순서로 준비하는 것이 가장 효율적이다.

1. GitHub 저장소 생성
2. Supabase 프로젝트 생성
3. Cloudflare Pages 프로젝트 생성
4. 로컬 개발 환경 준비
5. 환경변수 수집 및 정리
6. 데이터베이스 구조와 권한 정책 초안 준비
7. 배포 연결 확인

---

## 3. GitHub에서 준비할 사항

### 3.1 새 저장소 생성

반드시 새 GitHub 레포지토리를 하나 준비하는 것이 좋다.

권장 사항:

- 저장소 이름 예시: `lotto-rps-web`
- 공개 여부: `private` 권장
- 기본 브랜치: `main`

### 3.2 저장소 생성 시 확인할 것

- README 자동 생성 여부는 자유
- `.gitignore`는 나중에 프로젝트 생성 시 맞춰도 된다
- 라이선스는 없거나 추후 추가해도 된다

### 3.3 GitHub에서 미리 준비해두면 좋은 것

- GitHub 계정 로그인 상태 확인
- 로컬 PC에서 Git 사용 가능 상태 확인
- 원격 저장소 주소 확인
  - HTTPS 또는 SSH 중 하나 선택
- 협업할 사람이 있다면 미리 초대할지 결정

### 3.4 실제로 필요한 결과물

최소한 아래 2가지는 있어야 한다.

- 새 GitHub 레포지토리 생성 완료
- 로컬 프로젝트와 연결할 원격 저장소 URL 확보

---

## 4. Supabase에서 준비할 사항

### 4.1 새 프로젝트 생성

Supabase에서 새 프로젝트를 하나 만들어야 한다.

생성 시 필요한 항목:

- 프로젝트 이름
- 데이터베이스 비밀번호
- 리전 선택

권장:

- 프로젝트 이름은 GitHub 레포 이름과 유사하게 맞춘다
- DB 비밀번호는 별도 안전한 곳에 저장한다

### 4.2 프로젝트 생성 후 확보해야 할 값

Supabase 프로젝트가 생성되면 아래 값을 확인해둬야 한다.

- `Project URL`
- `anon public key`
- `service_role key`

이 값들은 이후 로컬 개발과 Cloudflare 배포 환경 모두에 필요하다.

### 4.3 Supabase에서 사전 준비해야 할 데이터 구조

최소한 아래 테이블을 만들 계획이어야 한다.

- `users`
- `sessions`
- `game_plays`
- `lotto_tickets`

각 테이블 역할:

- `users`: 닉네임과 PIN 해시 저장
- `sessions`: 로그인 세션 관리
- `game_plays`: 가위바위보 플레이 기록 저장
- `lotto_tickets`: 승리 시 발급되는 로또 번호 저장

### 4.4 Supabase에서 사전 준비해야 할 정책

개발 전에 `RLS(Row Level Security)`를 어떻게 적용할지 방향을 정해두는 것이 좋다.

필요한 정책 방향:

- 사용자는 본인 티켓만 조회 가능
- 사용자는 본인 플레이 기록만 조회 가능
- 본인 티켓만 `viewed` 처리 가능
- 세션 검증 없이 민감 데이터 접근 불가

### 4.5 Supabase에서 미리 확인해둘 것

- SQL Editor 사용 가능 여부
- Table Editor 접근 가능 여부
- API 키 복사 가능 여부
- 프로젝트 일시 중지 정책 또는 과금 조건 확인

### 4.6 실제로 필요한 결과물

- Supabase 프로젝트 생성 완료
- `URL`, `anon key`, `service role key` 확보
- DB 비밀번호 안전하게 보관

## 5. Cloudflare에서 준비할 사항

### 5.1 Cloudflare Pages 프로젝트 준비

Cloudflare에서 `Pages` 프로젝트를 사용할 예정이므로 계정과 프로젝트를 준비해야 한다.

필요한 준비:

- Cloudflare 계정 로그인 상태 확인
- Pages 사용 가능 여부 확인
- GitHub 연동 권한 허용 가능 상태 확인

### 5.2 Cloudflare Pages에 연결할 정보

아래 정보가 준비되어야 한다.

- 연결할 GitHub 저장소
- 배포 브랜치
- 프레임워크: `Next.js`

### 5.3 Cloudflare에 미리 준비해야 할 설정

- 환경변수 입력 위치 확인
- Preview / Production 배포 개념 확인
- 필요 시 커스텀 도메인 사용 여부 결정

### 5.4 실제로 필요한 결과물

- Cloudflare Pages 프로젝트 생성 가능 상태
- GitHub 저장소 연결 준비 완료
- 환경변수 입력 위치 확인 완료

---

## 6. 로컬 개발환경에서 준비할 사항

### 6.1 필수 설치 항목

개발 시작 전에 아래가 준비되어 있어야 한다.

- `Node.js`
- `npm` 또는 `pnpm`
- `Git`
- 코드 편집기
  - 예: `VS Code`

### 6.2 로컬에서 확인할 것

- 터미널에서 `node -v` 실행 가능
- 터미널에서 `npm -v` 또는 `pnpm -v` 실행 가능
- 터미널에서 `git --version` 실행 가능
- GitHub에 push 가능한 로그인 상태인지 확인

### 6.3 작업 폴더 준비

현재 작업 폴더가 이미 있다면 그대로 사용해도 된다.  
다만 아래는 확인해두는 것이 좋다.

- Git 저장소로 초기화할지 여부
- GitHub 원격 저장소 연결 여부
- `.env.local`을 둘 수 있는 구조인지 여부

### 6.4 실제로 필요한 결과물

- Node.js, 패키지 매니저, Git 설치 완료
- 로컬에서 GitHub push 가능 상태 확인

## 7. 환경변수로 준비할 값

개발 전에는 아래 값을 정리해두는 것이 중요하다.

### 7.1 로컬 `.env.local`에 들어갈 가능성이 높은 값

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- 세션 서명용 비밀값

### 7.2 주의사항

- `SUPABASE_SERVICE_ROLE_KEY`는 절대 클라이언트 코드에 노출되면 안 된다
- 서버 전용 비밀값은 Git에 커밋하면 안 된다
- `.env.local`은 버전관리에서 제외해야 한다

### 7.3 실제로 필요한 결과물

- 환경변수 목록 정리 완료
- 로컬용 값 확보
- 배포용 값 확보 가능 상태

## 8. 로그인 방식 관련 사전 결정 사항

이번 프로젝트는 `닉네임 + 4자리 PIN` 방식이므로 아래 사항을 미리 정해두면 개발이 빨라진다.

### 8.1 닉네임 규칙

정해둘 필요가 있는 항목:

- 최소 길이
- 최대 길이
- 허용 문자
- 중복 허용 여부

권장:

- 중복 불가
- 너무 긴 닉네임 제한

### 8.2 PIN 규칙

정해둘 필요가 있는 항목:

- 반드시 숫자 4자리만 허용할지
- 연속 실패 시 제한을 둘지
- 변경 기능이 필요한지

권장:

- 숫자 4자리만 허용
- 초기 버전은 변경 기능 없이 시작 가능
- 연속 실패 시 짧은 지연 또는 차단 고려

### 8.3 세션 규칙

정해둘 필요가 있는 항목:

- 로그인 유지 기간
- 브라우저 종료 후 유지 여부
- 로그아웃 시 세션 폐기 방식

---

## 9. 데이터베이스 설계 전에 정해둘 것

### 9.1 users

최소 확정이 필요한 항목:

- `nickname` 중복 불가
- `pin_hash` 저장 방식

### 9.2 sessions

최소 확정이 필요한 항목:

- 세션 토큰 저장 여부
- 만료 시간 저장 여부

### 9.3 game_plays

최소 확정이 필요한 항목:

- 사용자 선택값 저장
- 컴퓨터 선택값 저장
- 결과 저장
- 생성 시각 저장

### 9.4 lotto_tickets

최소 확정이 필요한 항목:

- 번호 6개 저장 방식
- 상태값 `issued`, `viewed`
- 발급 시각 저장
- 확인 시각 저장

## 10. 배포 전에 정해둘 운영 방식

### 10.1 브랜치 전략

최소한 아래 정도는 정해두는 것이 좋다.

- `main`: 배포 기준 브랜치
- 필요 시 `dev`: 개발 브랜치

### 10.2 배포 방식

정해둘 것:

- GitHub `main` push 시 자동 배포할지
- Preview 배포도 사용할지

권장:

- `main`은 배포 브랜치
- 기능 확인은 Preview 배포 활용

### 10.3 도메인

정해둘 것:

- 기본 `pages.dev` 도메인만 사용할지
- 커스텀 도메인을 붙일지
