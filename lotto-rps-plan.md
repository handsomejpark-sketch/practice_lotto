# 모바일 가위바위보 로또 티켓 웹앱 기획서

## 1. 개요

- 본 프로젝트는 모바일 우선 웹페이지 형태의 이벤트 게임 서비스다.
- 사용자는 `닉네임 + 4자리 비밀번호(PIN)`로 로그인 후 가위바위보 게임을 플레이할 수 있다.
- 사용자가 게임에서 승리하면 로또 티켓 1장을 지급한다.
- 지급된 로또 티켓은 사용자 계정에 누적 저장되며 `내 티켓` 화면에서 확인할 수 있다.
- 시스템 구성 기준은 `Next.js + Supabase + GitHub + Cloudflare Pages`다.

## 2. 서비스 목표

- 짧고 직관적인 게임 경험으로 모바일 사용자의 참여를 유도한다.
- 복잡한 소셜 로그인 없이 빠르게 입장할 수 있게 한다.
- 승리 보상으로 로또 티켓을 지급해 이벤트형 재미를 만든다.
- 티켓 누적 구조를 통해 재방문 동기를 만든다.

## 3. 핵심 사용자 흐름

1. 사용자가 모바일 웹페이지에 접속한다.
2. 랜딩 화면에서 이벤트 소개와 게임 시작 버튼을 확인한다.
3. 로그인하지 않은 경우 `닉네임 + 4자리 PIN`으로 회원가입 또는 로그인한다.
4. 로그인 후 가위, 바위, 보 중 하나를 선택해 게임을 시작한다.
5. 서버가 컴퓨터 선택값과 승패를 확정한다.
6. 승리한 경우 로또 티켓 1장을 생성해 사용자 계정에 저장한다.
7. 사용자는 `내 티켓` 화면에서 현재까지 획득한 티켓을 최신순으로 확인한다.
8. 사용자가 티켓을 확인하면 상태를 `viewed`로 변경할 수 있다.

## 4. 기술 아키텍처

### 4.1 프론트엔드

- 프레임워크: `Next.js`
- 형태: 모바일 우선 웹앱
- UI 방향: 이벤트 프로모션 페이지 느낌의 강한 헤드라인, 큰 버튼, 직관적인 결과 피드백

### 4.2 인증

- 인증 방식: 자체 로그인
- 로그인 수단: `닉네임 + 4자리 PIN`
- 세션은 앱 서버 기준으로 발급 및 유지한다.
- PIN은 평문 저장하지 않고 해시 처리 후 저장한다.

### 4.3 데이터 저장

- 데이터베이스: `Supabase Postgres`
- 주요 데이터는 사용자, 게임 플레이 기록, 로또 티켓, 세션 정보로 구성한다.
- 사용자 데이터 접근 제어는 `Supabase RLS(Row Level Security)`를 기준으로 한다.

### 4.4 배포 및 운영

- 원격 저장소: `GitHub`
- 배포 대상: `Cloudflare Pages`
- 기본 운영 흐름은 `로컬 개발 -> GitHub push/pull -> Cloudflare Pages 자동 배포`다.

- 로그인/회원가입 버튼

- 현재 보유 티켓 수

### 5.2 로그인 화면

- 회원가입 버튼
- 로그인 버튼

### 5.3 게임 화면

- 가위, 바위, 보 선택 버튼
- 사용자 선택값 표시
- 컴퓨터 선택값 표시
- 승/패/무 결과 표시
- 승리 시 발급된 로또 번호 표시
- 다시 도전하기 버튼

### 5.4 내 티켓 화면

- 최신순 티켓 목록
- 각 티켓의 로또 번호 6개
- 발급 시각
- 상태 배지
- 미확인 티켓 확인 처리

## 6. 기능 요구사항

### 6.1 회원가입/로그인

- 사용자는 `닉네임 + 4자리 PIN`으로 가입 및 로그인할 수 있다.
- 닉네임은 중복 불가다.
- PIN은 숫자 4자리만 허용한다.
- PIN은 서버에서 해시 처리 후 저장한다.
- 로그인 성공 시 세션을 발급한다.
- 로그인 실패 시 에러 메시지를 노출한다.

### 6.2 게임

- 게임은 1회 선택형 가위바위보 방식으로 진행한다.
- 사용자는 `rock`, `paper`, `scissors` 중 하나를 선택한다.
- 서버는 컴퓨터 선택값을 생성하고 최종 승패를 계산한다.
- 결과는 `win`, `lose`, `draw` 중 하나로 반환한다.
- 플레이 횟수 제한은 두지 않는다.

### 6.3 로또 티켓 지급

- 사용자가 승리하면 로또 티켓 1장을 생성한다.
- 번호는 `1~45` 범위에서 중복 없는 6개 숫자로 생성한다.
- 티켓은 사용자 계정에 저장된다.
- 패배 또는 무승부 시 티켓은 생성되지 않는다.

### 6.4 티켓 관리

- 티켓은 누적 보관형으로 관리한다.
- 티켓 상태는 `issued`, `viewed` 두 단계만 사용한다.
- 사용자가 티켓을 확인하면 `viewed` 상태로 변경할 수 있다.
- 티켓 목록은 최신순으로 조회한다.

### 6.5 보안 최소 요구사항

- PIN 평문 저장 금지
- 로그인 API에 기본적인 실패 응답 제어 적용
- 필요 시 로그인 실패 횟수 제한 또는 짧은 지연 적용
- 사용자는 본인 데이터만 조회 및 수정 가능해야 한다.

## 8. API 및 서버 동작 초안

### 8.1 회원가입 API

- 목적: 닉네임과 PIN으로 신규 사용자 생성
- 입력:
  - `nickname`
  - `pin`
- 처리:
  - 닉네임 중복 확인
  - PIN 형식 검증
  - PIN 해시 저장

### 8.2 로그인 API

- 목적: 사용자 인증 및 세션 발급
- 입력:
  - `nickname`
  - `pin`
- 출력:
  - 로그인 성공 여부
  - 사용자 기본 정보
  - 세션 정보

### 8.3 게임 결과 처리

- 목적: 가위바위보 결과 계산 및 승리 시 티켓 발급
- 입력:
  - `userChoice: "rock" | "paper" | "scissors"`
- 출력:
  - `computerChoice`
  - `result`
  - `awardedTicket` nullable

### 8.4 내 티켓 목록 조회

- 목적: 로그인 사용자의 티켓 목록 조회
- 출력:
  - `id`
  - `numbers`
  - `status`
  - `createdAt`
  - `viewedAt`

### 8.5 티켓 확인 처리

- 목적: 티켓 상태를 `issued`에서 `viewed`로 변경
- 입력:
  - `ticketId`

### 8.6 서버 신뢰 경계

- 게임 승패 계산은 클라이언트가 아니라 서버 기준으로 확정한다.
- 티켓 생성과 저장은 서버 권한으로만 처리한다.
- 사용자 인증 확인 후에만 게임 기록과 티켓 데이터에 접근할 수 있다.

## 9. 보안 및 권한 정책

- `Supabase RLS`를 적용해 사용자는 본인 `game_plays`, `lotto_tickets`만 조회 가능해야 한다.
- 세션 확인 후 본인 소유 데이터에 대해서만 상태 변경이 가능해야 한다.
- PIN은 해시로만 저장하고 원문은 보관하지 않는다.
- `SUPABASE_SERVICE_ROLE_KEY`는 서버 환경에서만 사용하고 클라이언트에 노출하지 않는다.
- 공개 클라이언트에는 `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`만 사용한다.

## 10. GitHub 및 배포 운영 기준

### 10.1 GitHub

- 프로젝트 소스의 기준 저장소는 GitHub다.
- 개발 중 변경사항은 GitHub 기준으로 push/pull 한다.
- 배포 가능한 상태의 코드는 GitHub 브랜치 기준으로 관리한다.

### 10.2 Cloudflare Pages

- `Next.js` 앱은 Cloudflare Pages에 배포한다.
- GitHub 저장소와 연결해 브랜치 반영 시 자동 배포되도록 구성한다.
- 배포 환경변수는 Cloudflare Pages 프로젝트 설정에서 관리한다.

### 10.3 환경변수

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- 세션 서명 또는 암호화에 필요한 서버 전용 비밀값

### 12.1 회원가입/로그인

- 닉네임 중복 가입 차단
- 4자리 숫자가 아닌 PIN 차단

- 잘못된 PIN으로 로그인 실패
- 세션 유지 및 로그아웃 확인

### 12.2 게임 로직

- 가위/바위/보 규칙 정확성 검증
- 무승부 처리 검증
- 패배 시 티켓 미생성 검증
- 승리 시 티켓 1장만 생성되는지 확인

### 12.3 티켓 데이터

- 로또 번호가 1~45 범위인지 확인
- 번호 6개가 중복되지 않는지 확인
- 티켓이 최신순으로 조회되는지 확인
- 확인 처리 후 상태가 `viewed`로 변경되는지 확인

### 12.4 권한 및 보안

- 다른 사용자의 티켓과 게임 기록이 조회되지 않는지 확인
- 본인 소유가 아닌 티켓 상태 변경이 차단되는지 확인
- PIN 평문이 저장되지 않는지 확인
- 서버 전용 키가 클라이언트 코드에 포함되지 않는지 확인

### 12.5 모바일 UI

- 360px 전후 화면 폭에서 레이아웃 깨짐이 없는지 확인
- 버튼 탭 영역이 충분한지 확인
- 결과 화면과 티켓 목록의 세로 스크롤 동작 확인

## 14. 최종 정리

- 이 서비스는 모바일 웹 기반 이벤트 게임이다.
- 사용자는 `닉네임 + 4자리 PIN`으로 간편 로그인한다.
- 로그인한 사용자가 가위바위보에서 이기면 로또 티켓을 1장 획득한다.
- 티켓은 `Supabase`에 누적 저장되며 `내 티켓` 화면에서 관리한다.
- 소스 관리는 `GitHub`, 배포는 `Cloudflare Pages`를 기준으로 운영한다.
- 초기 버전은 빠른 참여 경험과 단순한 인증 흐름에 집중한다.
